# <a name="c26437-dont_slice"></a>C26437 DONT_SLICE

"Ne tranchez pas."

**Lignes directrices de base**de CMD : [ES.63 : Ne tranchez pas](https://github.com/isocpp/CppCoreGuidelines/blob/master/CppCoreGuidelines.md#Res-slice)

[Le tranchage](https://en.wikipedia.org/wiki/Object_slicing) est autorisé par le compilateur et peut être considéré comme un cas spécial de fonte implicite dangereuse. Même si elle est faite intentionnellement et ne conduit pas à des problèmes immédiats, il est encore fortement découragé car il rend le code plutôt intenable en forçant des exigences supplémentaires sur les types de données connexes. Cela est particulièrement vrai si les types sont polymorphes ou impliquent la gestion des ressources.

## <a name="remarks"></a>Notes

- Cette règle avertirait non seulement sur les affectations explicites, mais aussi sur le tranchage implicite qui se produit lorsque le résultat est retourné de la fonction actuelle ou des données transmises comme arguments à d’autres fonctions.
  - Les avertissements signaleraient également les cas où l’affectation n’implique pas de découpe de données réelles (p. ex., si les types sont vides ou ne font aucune manipulation de données dangereuses). Ces avertissements devraient encore être adressés pour éviter toute régression indésirable si les données types ou les changements de comportement à l’avenir.

## <a name="example"></a>Exemple

les points de tranchage à obsolètes

```cpp
interface
struct id {
    int value;
};

struct id_ex : id {
    int extension;
};

bool read_id(stream &s, id &v) {
    id_ex tmp{};
    if (!s.read(tmp.value) || !s.read(tmp.extension))
        return false;

    v = tmp; // C26437
    return true;
}
```

les points de tranchage à obsolète, interface - corrigé

```cpp
// ...
bool read_id(stream &s, id_ex &v) {
// ...
```
